# ADR-0026: Material picker URL scheme

**Status:** Accepted

**Date:** 2026-05-21

## Context

Two parallel material-add UIs lived side-by-side in the checkout app:

1. The **picker** (Sheet overlay opened from per-workshop "+ Material
   hinzufügen" buttons on `/visit`) — controlled by local React state,
   scoped to one workshop.
2. The **legacy one-off page** at `/material/add?priceList=<id>` —
   reached only from pricelist-PDF QR codes generated by
   `functions/src/price_list/get_price_list_pdf_url.ts`. A two-stage
   list→item form, separate code path, separate auth layout.

Issue #213 asked to retire the legacy page and use the same picker for
pricelist QR scans (and any future per-item stickers). The picker also
needs to grow an embedded QR scanner in its search box, so the
mechanism that opens it must be uniform across entry points — button
click, QR scan, deep link.

## Decision

The picker is **always** opened via a URL route. Four nested sub-routes
of `/visit` each render the picker with a specific scope:

| Entry point | URL | Scope |
| --- | --- | --- |
| Generic "add anything" (QR scanner empty-state) | `/visit/add` | `all` — every `active && userCanAdd` catalog item |
| Workshop "+" button | `/visit/add/workshop/$workshopId` | `workshop` — items where `workshops` array-contains the id |
| Pricelist QR (PDF) | `/visit/add/list/$listId` | `list` — items whose doc id ∈ `priceList.items[]` |
| Per-item QR / sticker (future) | `/visit/add/item/$code` | `item` — single item resolved by `code`, `variants[0]` pre-selected |
| Per-variant QR / sticker (future) | `/visit/add/item/$code/$variantId` | `item` — same item lookup; the named variant is pre-selected. Unknown variantIds fall back to `variants[0]` silently. |

All path segments are English (per house style: German only in UI labels).
The `/add` segment is retained — it disambiguates "open the add
overlay" from a hypothetical "view this workshop" page and matches the
conventional client-side modal-route shape.

### Picker scope model

`MaterialPicker` takes a discriminated `PickerScope`:

```ts
type PickerScope =
  | { kind: "all" }
  | { kind: "workshop"; workshopId; workshopLabel }
  | { kind: "list"; listId; listName }
  | { kind: "item"; code; itemId; variantId? }
```

The host route loads the appropriate catalog slice (workshop query,
pricelist `documentId() in […]` query, or single-item `where("code","==",…)`
query) and passes pre-filtered items to the picker. The picker itself
no longer narrows by workshop. In `item` scope the variant chooser
auto-expands so the user lands directly on the form.

### Workshop attribution

For non-workshop scopes, each catalog item has a `workshops: string[]`
array. The route picks the workshop the item is added to with an
**overlap-first** rule:

1. If `item.workshops` intersects `checkout.workshopsVisited`, route
   the add to the overlapping workshop and leave `workshopsVisited`
   untouched.
2. Otherwise, route to `item.workshops[0]` **and** `arrayUnion` that
   value into `checkout.workshopsVisited` in the same write so the
   dashboard section and checkbox appear without a second user action.

### Catalog `code` uniqueness

The `/visit/add/item/$code` route resolves by `code` (the SKU). Codes
are human-readable labels that may need to change, so they stay as a
field — not the Firestore document id — to keep renames cheap. Uniqueness
is enforced server-side:

- Catalog writes flow through the new `upsertCatalogItem` callable in
  `functions/src/catalog/`. Its transaction queries
  `catalog where code == newCode limit 2`, rejects with `already-exists`
  if any returned doc has a different id than the one being written,
  then sets the doc.
- `firestore/firestore.rules` denies all client-side writes to `catalog`
  (`allow write: if false`). The callable bypasses rules via Admin SDK.
  Reads stay open so the picker can query the collection directly.
- A one-shot script `scripts/find-duplicate-catalog-codes.ts` surfaces
  any legacy duplicates so an admin can disambiguate before they block
  edits.

### Legacy route removed

`/material/add?priceList=<id>` is deleted. Pricelist QRs now encode
`https://<CHECKOUT_DOMAIN>/visit/add/list/<priceListId>`. The PDF
content hash already includes `qrUrl`, so cached PDFs invalidate on the
next admin-side regeneration.

## Consequences

**Pros:**
- Single entry mechanism for the picker: button, QR scan, deep link,
  and browser back/forward all go through the same router code.
- Sub-route URLs are self-describing — a future QR scanner just injects
  the scanned URL; no parsing of opaque payload formats.
- The dashboard stays mounted behind the Sheet (via `<Outlet />`), so
  closing the picker is plain `navigate({ to: "/visit" })` with full
  history.
- Catalog uniqueness is enforced where it can actually be checked
  (transaction over a collection query), not where it can't (rules).

**Cons:**
- Catalog writes now require a callable round-trip; offline-first
  workflows aren't supported for catalog edits. Acceptable — admin
  is online-only.
- One indirection in the admin UI (`useAsyncMutation` +
  `httpsCallable`) instead of a direct `addDoc`/`updateDoc`.

**Tradeoffs:**
- *Search params (`?add=list:<id>`)* rather than sub-routes. Rejected:
  needs custom parsing, weaker type inference, no nested layout boundary
  for the loader. Sub-routes are TanStack Router's natural shape.
- *Per-item doc id = code* to enforce uniqueness natively. Rejected:
  forces a doc-id migration on every code rename, touching
  `priceLists.items` and historical `checkoutItems.catalogId`
  DocumentReferences. Callable + transaction is the smaller change.
- *Permanent 301 redirect from `/material/add?priceList=…`* to the new
  URL. Rejected per issue #213: existing printed materials are
  re-generated as part of the rollout.
