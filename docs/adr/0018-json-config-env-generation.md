# ADR-0018: JSON Config with Generated Env Files

**Status:** Proposed

**Date:** 2026-03-22

**Applies to:** `scripts/generate-env.ts`, `functions/.env*`, `web/apps/checkout/.env*`, `web/apps/admin/.env*`, `maco_gateway/.env.local`, `scripts/.env`

## Context

1. **Canonical config in one place.** Hardcoded values scattered across env files, npm scripts, and source code lead to subtle inconsistencies and confusing test failures. Every configurable value should have a single authoritative definition, even if it rarely changes.

2. **Private operations repo.** Operational documentation (deployment procedures, monitoring links, incident runbooks) should live alongside the config values they reference, in a repo that isn't publicly indexed. While not a security boundary, keeping explicit service URLs and infrastructure details out of a public repo avoids making automated reconnaissance trivial.

## Decision

Replace committed symlinks with a **JSON config + TypeScript generator** approach:

### Operations repo provides structured config

- `config.jsonc` — production values in a nested JSON structure (Firebase settings, web deployment config, function parameters).
- `config.local.jsonc` — local development overrides (emulator URLs, test secrets). Deep-merged on top of `config.jsonc`.

### This repo generates env files

`scripts/generate-env.ts` reads the JSON configs and generates all `.env` files that each component needs. The mapping from JSON paths to env variable names is defined declaratively in the script.

Generated files:

| File | Source | Purpose |
|------|--------|---------|
| `functions/.env.local` | local (merged) | Emulator: params + test secrets |
| `functions/.env.<projectId>` | production | Deploy: params only (secrets via Secret Manager) |
| `web/apps/checkout/.env.development` | local (merged) | Checkout Vite dev server |
| `web/apps/checkout/.env.production` | production | Checkout Vite production build |
| `web/apps/admin/.env.development` | local (merged) | Admin Vite dev server |
| `web/apps/admin/.env.production` | production | Admin Vite production build |
| `maco_gateway/.env.local` | local (merged) | Local gateway development |
| `scripts/.env` | production | Utility scripts |
| `.firebaserc` | production | Firebase project ID |

### Generated files are gitignored

All generated `.env` files are in `.gitignore`. The generation step runs as part of `dev.sh` and `npm run generate-env`.

## Consequences

**Pros:**
- Single source of truth for all config values — no more duplicating variables across files
- Firebase CLI works correctly (reads real files, not symlinks)
- Clean local override mechanism via `config.local.jsonc` deep merge
- Declarative mapping makes it obvious which JSON field maps to which env var
- Generated header (`# Generated by scripts/generate-env.ts -- do not edit`) prevents accidental manual edits

**Cons:**
- Extra generation step required before first run (automated in `dev.sh`)
- Operations repo must migrate from flat `.env` files to `config.jsonc` format
- Adding a new env variable requires editing `generate-env.ts` (mapping) and the operations repo JSON schema

**Alternatives considered:**
- **Fix symlink handling in Firebase CLI**: Not feasible — the validation logic is in the Firebase tooling, not under our control.
- **Copy files instead of symlinks**: Loses the live-update benefit and still requires duplication across files in the operations repo.
- **dotenv-vault or similar tools**: Adds external service dependency for a simple configuration problem.
